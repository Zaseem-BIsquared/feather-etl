## Summary

**Symptom:** Five issues found during Slice 2-3 review (docs/reviews/2026-03-27-slice2-3-review.md): H-1 (watermark nulling on touch-skip), H-2 (negative overlap_window_minutes), M-1 (raw SQL interpolation), M-2 (connection leak in state.py), L-1 (duplicate validation JSON write).

**Trigger:** Review of Slice 2 (committed at 239429e) + Slice 3 (uncommitted worktree, now lost).

**Root Cause:** Multiple independent issues — each traced below.

## Investigation

### Key discovery: Slice 3 code is not on main

The review was conducted against both Slice 2 (on main) and Slice 3 (in an uncommitted worktree). The Slice 3 worktree has been cleaned up — only the plan file (`docs/plans/2026-03-27-slice3-incremental-extraction.md`) exists on main, not the implementation code.

This means H-1 and H-2 reference behavior that would only manifest when Slice 3 code is added. However, both can be fixed proactively now:

### H-1 — Touch-file skip path clears incremental watermark

- **Root cause:** `state.py:154-160` — `write_watermark()` UPDATE statement overwrites all fields unconditionally. When the touch-skip path (pipeline.py:60-66) calls it without `last_value`, any stored watermark value would be nulled.
- **Current impact:** None — `write_watermark()` doesn't touch `last_value` column yet. But when Slice 3 adds `last_value` support, the unconditional UPDATE pattern will cause this bug.
- **Fix:** Make `write_watermark()` use conditional UPDATE — only set fields that are explicitly provided. Use COALESCE in SQL to preserve existing values.
- **Confidence:** High — traced the exact SQL statements.

### H-2 — overlap_window_minutes missing validation

- **Root cause:** `config.py:237` — `overlap_window_minutes` is read directly from YAML with no validation. No check in `_validate()` for negative values.
- **Current impact:** Low — pipeline doesn't use this field yet. But negative values would cause missed rows when Slice 3 incremental extraction is implemented.
- **Fix:** Add `>= 0` validation in `_validate()`.
- **Confidence:** High — traced config loading path.

### M-1 — Raw SQL composition from config

- **Root cause:** `duckdb_file.py:79` — `f"SELECT * FROM source_db.{table}"` interpolates the table name directly. Config already validates `target_table` names (config.py:164-175) but `source_table` gets no validation.
- **Current impact:** Medium — source_table comes from trusted YAML config, but malformed values break queries and hostile config could execute unintended SQL.
- **Fix:** Validate `source_table` format during config validation (must match `schema.table` pattern for DuckDB sources, or be a valid filename for CSV/SQLite).
- **Confidence:** High — CSV and SQLite sources already use parameterized queries, only DuckDB source interpolates.

### M-2 — State DB connection leak on exception paths

- **Root cause:** `state.py:127-226` — `read_watermark()`, `write_watermark()`, `record_run()`, `get_status()`, and `init_state()` all call `con.close()` on the normal path but not in `finally` blocks. Exceptions before close leak handles.
- **Fix:** Wrap each method in `try/finally` to guarantee `con.close()`.
- **Confidence:** High — directly visible in code.

### L-1 — Duplicate validation JSON write

- **Root cause:** `cli.py:18` calls `write_validation_json()` in `_load_and_validate()`, then `pipeline.py:124` calls it again in `run_all()`. The `feather run` command calls both.
- **Fix:** Remove the call from `run_all()` — let the CLI layer own validation artifact writing.
- **Confidence:** High — traced both call sites.
